# Feature Layer

## Responsibility

The feature layer turns stored market data into model-ready inputs.

It owns:
- feature specifications
- visible-as-of bar aggregation
- feature builders
- feature normalization
- feature datasets
- per-bar progress and one-row-per-decision time context
- editable indicator registry and compact training stores

It must keep generated model inputs close to ratios, positions, continuous
indicator states, time-position features, and explicit market tradability
constraints. Raw prices, cash amounts, share counts, and volumes may be used
internally by builders and environments, but they should not be emitted as
model input columns.

Portfolio/account fields are a contract only. They are dynamic environment
state and must not be pre-generated into feature datasets.

## v1 Contract

The first stable contract is `feature_v1`, exposed as
`feature_layer.DEFAULT_FEATURE_SPEC`.

Core rules:
- `5min` is the preferred base frequency and trade frequency.
- `15min`, `30min`, and `60min` states are derived from visible completed
  `5min` bars.
- Official `daily` bars remain the completed daily reference. The current
  unfinished daily state is built from visible intraday bars only for the
  current decision moment.
- `weekly` and `monthly` completed states are derived from official `daily`
  bars. The current unfinished week/month may include the current partial daily
  bar built from visible intraday data.
- `monthly` is available in `feature_v1` as an optional long-horizon sequence.
  It is disabled by default in the dataset builder but can be enabled when the
  Agent contract includes monthly inputs.
- Higher-frequency states may include the current partial bar, but only from
  data with `available_at <= decision_time`.
- The model has two decision stages:
  - `open_auction`: one decision at the daily open using the opening price and
    previous visible history, without the first completed 5min bar.
  - `bar_close`: one decision after each completed 5min bar, executed at that
    bar close in the first simulator contract.
- Daily turnover is useful, but intraday decisions only receive visible
  historical turnover features such as `turn_lag1`, `turn_ma20`, and `turn_z20`.
- Market tradability is generated by this layer as `market_can_buy`,
  `market_can_sell`, `is_tradeable`, `is_limit_up`, `is_limit_down`, and
  `is_zero_volume`.
- Account-dependent tradability such as `can_buy`, `can_sell`,
  `available_position_ratio`, `max_buy_value_ratio`, and
  `max_sell_value_ratio` is provided later by the environment layer.
- Full training data is stored as per-symbol parquet parts under `feature_parts/`,
  with `feature_parts_manifest.json` and `model_input_contract.json` as the
  Agent-facing metadata. CSV files are bounded review samples only; Feature Layer
  does not write `portfolio.csv`.

The default technical indicators are intentionally limited:
- MACD as continuous close-scaled values
- KD as normalized continuous values
- EFI2/EFI13 as normalized continuous values, calculated from `EMA((close - REF(close, 1)) * volume, 2/13)`
- EMA channels: `EMA13/EMA21` and `EMA144/EMA169`

The longest v1 maturity target is 169 bars, checked independently for every
enabled frequency and symbol. It is not a build gate: a symbol with only three
bars is still usable. EMA169 keeps its fixed span and is recursively calculated
from those three real bars; it is never redefined as EMA3.

Two continuous availability features tell the model how mature the input is:
- `history_coverage = min(available bars / 169, 1)`
- `sequence_valid_ratio = min(real sequence rows / configured window, 1)`

`decision_context` contains one row per decision: intraday slot, day progress,
session flags, minutes to close, and open-auction stage. These fields are not
repeated in every market frequency sequence. Per-bar `progress` remains in the
market stream because it describes a partial higher-period bar.

The compact store keeps unique `market_bars` and `market_features`, plus a
small `decision_index` and current-period `decision_snapshots`. It does not
repeat every historical window for every decision. `load_compact_market_sequence`
resolves an indexed window, left-pads zeros, and returns `sequence_mask`, where
`0` means padding and `1` means a real row. Preflight reports incomplete
maturity as a warning, not an error.

Stored derived bars use downloaded canonical bases: `5min -> 15min`,
`30min -> 60min`, and `daily -> weekly/monthly`. Materialization aggregates
the selected universe directly inside DuckDB and atomically replaces one target
frequency at a time. It does not copy multi-million-row histories into Pandas.
Progress follows the active target transaction, and cancellation is honored
before a target is committed.

`derived_bar_manifest` records each symbol's base range, row count, content
fingerprint, target row count, and generation time. An unchanged slice is
reported as `skipped`; a pure append rebuilds only the previous active day,
week, or month plus new data. A historical correction changes the fingerprint
and triggers a full rebuild for that symbol. Existing `derived_*` rows with
complete coverage can initialize the manifest without being regenerated.

Indicator configuration lives in `config/features/indicators.json`. Supported
indicator kinds are `macd`, `kd`, `efi`, and `ema_channel`. Each instance has a
stable ID, editable parameters, enabled frequencies, an `enabled` flag, a
validated `main_overlay` or `sub_panel` render target, and a default visibility
setting. Render visibility does not change whether an enabled indicator is part
of the model feature contract.
Saving the configuration changes the actual generated model columns. Arbitrary
Python expressions are deliberately not accepted.

Model input selection lives in `config/features/model_input.json` and is edited
through the Feature page's Model Input Blueprint. The Blueprint is an ordered
document with three item types:
- `group` and `comment` organize the document for people and do not affect the
  model schema.
- `feature` selects one channel, its input stream, enabled state, and, for
  market sequences, the frequencies where it is present.

The three model streams are:
- `market_sequence`: one padded `[window, channels]` tensor per frequency.
- `decision_context`: one ordered vector for decision-time and market-rule
  context.
- `runtime_state`: an ordered contract filled by the future trading environment
  at runtime rather than pre-generated into the dataset.

Feature rows are channel order: moving a row changes the corresponding tensor
column and schema hash. Disabling a row removes it from Loader output. The
compact Feature Store still retains all generated feature columns so changing a
Blueprint does not discard the underlying feature data. Every built store and
manifest embeds the compiled Blueprint, channel names, shapes, schema version,
and schema hash. This keeps an old training dataset reproducible even after the
editable project configuration changes.

Technical chart values use the same indicator calculation functions as model
feature generation. Visualization requests include hidden historical warm-up
rows and return only the requested aligned window. Non-technical market,
constraint, portfolio, and environment fields are never included in the
technical visualization response.

ST is a dated market-rule fact, not a model feature and not a permanent symbol
label. On an ST day, the effective price limit is 5% and
`market_can_buy = 0`; a legal exit remains available when volume and the
limit-down rule allow it. Non-ST days use the symbol board limit. The same
previous trading-day close is the limit reference for every intraday decision
in a session. A first-history session without that reference is warm-up only.

Missing `stock_status_daily` coverage is an error because the rule cannot be
reconstructed safely. Preflight still reports the real K-line inventory and
identifies the missing symbol-days instead of treating the market data as
empty.

Before Agent training, use the Agent preflight against the completed feature
dataset directory. The required runtime entry is `feature_parts/` plus
`feature_parts_manifest.json` and `model_input_contract.json`; no compact
`feature_store.duckdb` export is required.

## Allowed Dependencies

The feature layer may read from the data layer and may write generated feature
or dataset files into `runtime_layer`.

## Boundaries

It must not download raw data, own the canonical raw K-line database, implement
trading environments, execute arbitrary user code, or render charts. It may
prepare chart-ready technical series; actual rendering remains in Visualization.
